## 零、背景與問題盤點

### 0.1 現況

`GA4InsightsService.get_channels()`(`insights_service.py:563-583`)用「開發(`firstUserDefaultChannelGroup` conversions) / 收單(`sessionDefaultChannelGroup` conversions)」的比例,套固定門檻(>1.3 助攻型、<0.7 主攻型、其餘均衡)貼標籤,前端 `ChannelsTab.jsx` 顯示為「收單(最後接觸)」vs「開發(首次接觸)」對照表。

### 0.2 問題一:小樣本比例失真

樣本數很小時,比例會被單一事件推向極端。實測案例(2026-07-15,HUKUROU property):Email 渠道開發0/收單6,比例=0.00,被貼上「主攻型」——但只有 6 筆轉換,任何一筆歸屬差異都會讓比例整個翻轉,統計上不具代表性。現有邏輯只在「收單=0」時才判定「資料不足」,沒有考慮**總量**過小的情況。

### 0.3 問題二(已查證為事實,非假設):「收單」欄位文案有誤導

2026-07-15 查證使用者其中一個 GA4 property(HUKUROU)的「歸因設定 → 報表歸因模式」為**「以數據為準(Data-driven)」**,而非最後點擊。這代表 `sessionDefaultChannelGroup` 算出來的「收單」conversions,**已經是 GA4 資料驅動模型把功勞分散到路徑上多個接觸點之後的加權數字**,不是單純的 last-click 硬計數。而「開發」(`firstUserDefaultChannelGroup`)仍是硬計數(使用者一生中第一次造訪的管道,二元、不加權)。

現有文案「收單(最後接觸)」明確寫成 last-touch,會誤導使用者以為這是最後一次點擊的計數;實際上(在 Data-driven 屬性上)它已經內含路徑加權功勞,語意更接近「這個管道在所有轉換路徑中拿到的功勞總和」。

### 0.4 問題三:歸因模式是 per-property 設定,系統目前不知道、也沒告訴使用者

「報表歸因模式」在 GA4 後台是逐 property 設定的,不同客戶帳號可能設成 Last-click 而非 Data-driven。系統目前完全不查、不顯示這個設定,使用者除非自己去 GA4 後台手動查,否則不知道自己正在看哪種口徑的「收單」數字,也就無法正確解讀助攻/主攻標籤的意義。

### 0.5 範圍界定

本計劃**不**重建使用者層級的轉換路徑、**不**做 GA4 UI 那種前期/中期/後期接觸點精確功勞分配(該功能需要 BigQuery Export,docs/21 已列為 Non-goals、另案評估)。本計劃只在現有「兩點式近似」(開發 vs 收單)架構內做三件事:讓比例計算對小樣本更穩健、讓文案準確反映數字的真實意義、讓使用者能看到自己資料所依據的歸因模式。

### 0.6 問題四(第一~三波部署後發現,2026-07-15):標籤只反映「角色」,沒反映「量級」,容易誤讀成「主要訂單來源」

三波修正上線後,使用者用真實資料核對時發現:Email(開發9/收單45,比例0.20)、Mobile Push Notifications(開發8/收單17,比例0.47)都被標「主攻型」——這是公式正確運作的結果(這兩個渠道本來就幾乎不會是任何人的第一次接觸,是喚回性質的渠道,開發數字天生就小),不是 bug,第三波的樣本量門檻(≥10)也正確判斷這兩者樣本量足夠、不該歸類為「資料不足」。

但問題在於:「主攻型」這三個字在語意上容易被讀成「主要的/重要的管道」,而 Email(收單45)、Push(收單17)跟 Organic Social(收單2654)、Cross-network(收單2168)這種真正撐起訂單量的管道比起來是零頭。目前的表格沒有任何欄位或排序凸顯這個量級落差(渠道是依名稱字母序排列,不是依量級),使用者掃過一眼標籤欄,容易誤以為 Email/Push 才是主要訂單來源。

核心是「角色標籤」(assist/close/balanced,講的是渠道自身開發:收單的內部比例)與「量級/重要性」(這個渠道在全站訂單裡佔多少份量)是兩件事,現有 UI 把兩者混在一起看,沒有拆開。

## 四、第四波:量級與角色標籤脫鉤(2026-07-15 部署後追加)

**目標**:讓使用者一眼就能分辨「這個標籤講的是角色,不是重要性」,不用改動角色標籤(assist/close/balanced)的計算公式本身。

### 任務 4.1 — 表格/圖表依量級排序

**變更檔案**:`backend/modules/ga4/insights_service.py`(`get_channels()`)、`backend/tests/test_ga4_channel_dimension.py`

**問題**:`channels = sorted(set(session_by_channel) | set(first_user_by_channel))`(現行程式碼)是依渠道**名稱字母序**排序;只有在 `truncated`(> 20 筆)時才會額外依「開發+收單」總量重排並截斷。未截斷的情況下(多數帳號的預設維度渠道數 < 20),表格與圖表都維持字母序,量級大的渠道(Organic Social、Cross-network)跟量級小的渠道(Email、Mobile Push Notifications)交錯排列,使用者掃視表格時抓不到量級落差。

**實作步驟**:
1. 把「依（開發+收單）總量降冪排序」從現行只在 `truncated` 分支內做的邏輯,改成**無論是否截斷都套用**;也就是先排序,再視總數決定要不要截斷取前 20 筆(排序邏輯只寫一次,不要分成截斷/未截斷兩套)。
2. 圖表(`BarChart layout="vertical"`)直接沿用 `channels` 陣列順序,不用額外處理;表格 `<tbody>` 同理。

**驗收標準**:未截斷情境下(例如預設維度渠道通常 < 10 筆),`channels` 陣列已由(開發+收單)總量降冪排列,不再是字母序;截斷情境的既有行為(top 20、`truncated`/`total_row_count` 標記)不變;既有排序相關測試(`test_get_channels_truncates_to_top_20_by_combined_conversions`)需確認在新排序邏輯下仍然通過,若該測試原本依賴字母序的隱含假設要一併檢查更新。

**風險/回滾**:純顯示順序調整,不影響任何數字/標籤/比例計算,回歸風險低;單獨回滾(改回只在截斷分支排序)。

**驗收結果(2026-07-15 完成)**:
- `get_channels()` 排序邏輯只寫一次:`rows = sorted(rows, key=lambda r: r["closing_conversions"] + r["assisting_conversions"], reverse=True)` 移到截斷判斷之前、無條件執行,截斷分支改成單純 `rows[:CHANNEL_TOP_N]` 切片,不再重複排序。
- 既有 `test_get_channels_truncates_to_top_20_by_combined_conversions` 未依賴字母序假設,無需修改即通過。
- 新增測試 `test_get_channels_always_sorted_by_combined_conversions_even_without_truncation`:3 個渠道字母序(Email/Organic Social/Zzz Referral)刻意跟量級序相反,驗證未截斷情境下輸出順序已是量級降冪,不是字母序。

### 任務 4.2 — 標籤旁補「佔全站收單比例」

**變更檔案**:`backend/modules/ga4/insights_service.py`(`get_channels()`)、`frontend/src/components/GA4Insights/ChannelsTab.jsx`、`frontend/src/components/GA4Insights/GA4InsightsShared.jsx`

**問題**:光靠排序只能解決「掃視表格時的第一印象」,標籤本身(「主攻型」三個字)仍然沒有攜帶任何量級資訊,使用者若只看某一列、不看全表也可能誤讀。需要在標籤旁直接放量級佐證。

**實作步驟**:
1. `get_channels()` payload 新增 `total_closing_conversions`(**全部**渠道的收單加總,含被 top-20 截斷掉的那些,不能只加已回傳的 20 筆——否則截斷情境下佔比分母會失真)。
2. 前端在每列標籤(或 badge 旁)加小字顯示 `closing_conversions / total_closing_conversions` 的百分比,例如「主攻型(佔收單 0.8%)」;`fmtPct`(`GA4InsightsShared.jsx` 既有工具函式)可直接複用,不用新寫格式化邏輯。
3. 卡片說明文字補一句話講清楚兩件事的差異:「標籤講的是這個渠道『開發 vs 收單』的內部角色,不是它在全站訂單裡的重要性——量級請看佔比欄位」。

**驗收標準**:Email(收單45)、Mobile Push(收單17)這類小量渠道能清楚看到自己佔比是個位數百分比甚至更低;大渠道(Organic Social 等)佔比正確反映其真實份量;截斷情境下佔比分母仍是全量、不是只算被顯示的 20 筆(需要測試覆蓋這個邊界,否則容易埋隱藏 bug)。

**風險/回滾**:純新增欄位與顯示,不影響既有標籤計算;`total_closing_conversions` 若忘記排除 truncate 影響會導致佔比算錯,是本任務最大的正確性風險,測試要專門覆蓋這個情境;單獨回滾。

**驗收結果(2026-07-15 完成)**:
- `get_channels()` 在排序**前**先算 `total_closing_conversions = sum(r["closing_conversions"] for r in rows)`(此時 `rows` 還是全量、尚未截斷),payload 新增此欄位。
- 前端 `ChannelsTab.jsx` 標籤 badge 旁補一段小字「(佔收單 X%)」,用既有 `fmtPct()` 格式化,`title` 帶完整說明;`total_closing_conversions` 為 0 時不顯示(避免除以零)。
- 卡片說明文字新增固定顯示的一行(不依賴 `attribution_model` 是否可判斷),講清楚「標籤講的是內部角色、不是全站重要性」。
- 新增 2 項測試:`total_closing_conversions` 在未截斷情境正確加總、在截斷情境(25 筆截到 20 筆)仍是全部 25 筆的加總而非截斷後 20 筆的加總(專門驗證「分母必須是全量」這個正確性風險點)。
- 驗證:`python -m py_compile` 通過;`npm run build` 通過;GA4 相關 pytest 108 passed(含既有渠道對照/insights 測試,無回歸)。
