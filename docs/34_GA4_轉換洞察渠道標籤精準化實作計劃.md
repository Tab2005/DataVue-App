## 零、背景與問題盤點

### 0.1 現況

`GA4InsightsService.get_channels()`(`insights_service.py:563-583`)用「開發(`firstUserDefaultChannelGroup` conversions) / 收單(`sessionDefaultChannelGroup` conversions)」的比例,套固定門檻(>1.3 助攻型、<0.7 主攻型、其餘均衡)貼標籤,前端 `ChannelsTab.jsx` 顯示為「收單(最後接觸)」vs「開發(首次接觸)」對照表。

### 0.2 問題一:小樣本比例失真

樣本數很小時,比例會被單一事件推向極端。實測案例(2026-07-15,HUKUROU property):Email 渠道開發0/收單6,比例=0.00,被貼上「主攻型」——但只有 6 筆轉換,任何一筆歸屬差異都會讓比例整個翻轉,統計上不具代表性。現有邏輯只在「收單=0」時才判定「資料不足」,沒有考慮**總量**過小的情況。

### 0.3 問題二(已查證為事實,非假設):「收單」欄位文案有誤導

2026-07-15 查證使用者其中一個 GA4 property(HUKUROU)的「歸因設定 → 報表歸因模式」為**「以數據為準(Data-driven)」**,而非最後點擊。這代表 `sessionDefaultChannelGroup` 算出來的「收單」conversions,**已經是 GA4 資料驅動模型把功勞分散到路徑上多個接觸點之後的加權數字**,不是單純的 last-click 硬計數。而「開發」(`firstUserDefaultChannelGroup`)仍是硬計數(使用者一生中第一次造訪的管道,二元、不加權)。

現有文案「收單(最後接觸)」明確寫成 last-touch,會誤導使用者以為這是最後一次點擊的計數;實際上(在 Data-driven 屬性上)它已經內含路徑加權功勞,語意更接近「這個管道在所有轉換路徑中拿到的功勞總和」。

### 0.4 問題三:歸因模式是 per-property 設定,系統目前不知道、也沒告訴使用者

「報表歸因模式」在 GA4 後台是逐 property 設定的,不同客戶帳號可能設成 Last-click 而非 Data-driven。系統目前完全不查、不顯示這個設定,使用者除非自己去 GA4 後台手動查,否則不知道自己正在看哪種口徑的「收單」數字,也就無法正確解讀助攻/主攻標籤的意義。

### 0.5 範圍界定

本計劃**不**重建使用者層級的轉換路徑、**不**做 GA4 UI 那種前期/中期/後期接觸點精確功勞分配(該功能需要 BigQuery Export,docs/21 已列為 Non-goals、另案評估)。本計劃只在現有「兩點式近似」(開發 vs 收單)架構內做三件事:讓比例計算對小樣本更穩健、讓文案準確反映數字的真實意義、讓使用者能看到自己資料所依據的歸因模式。

## 一、第一波:歸因模式揭露(基礎建設,建議優先做)

### 任務 1.1 — 後端:唯讀查詢 GA4 property 的歸因設定

**變更檔案**:`backend/modules/ga4/client.py`、`backend/modules/ga4/insights_service.py`、`backend/modules/ga4/router.py`(或現有 insights router)、`backend/tests/test_ga4_channel_dimension.py`(或新測試檔)

**實作步驟**:
1. `GA4Client` 新增 `get_attribution_settings(user, property_id, db) -> tuple[dict | None, str | None]`:用既有 `get_credentials()` 取得的 `AnalyticsAdminServiceClient`,呼叫 `get_attribution_settings(name=f"properties/{property_id}/attributionSettings")`(唯讀,`analytics.readonly` scope 已足夠,不需新增 OAuth scope)。回傳 `reporting_attribution_model`(如 `CROSS_CHANNEL_DATA_DRIVEN` / `CROSS_CHANNEL_LAST_CLICK_MODEL` / `ADS_PREFERRED_LAST_CLICK` 等 enum)。
2. `insights_service.py` 的 `get_channels()` 呼叫上述方法,結果比照既有 snapshot 快取機制,以 `kind="attribution_settings"` 存入 `ga4_insights_snapshots`(重用 `repository.upsert_snapshot`/`get_latest_snapshot`,不新增資料表);快取有效期 24 小時(這個設定極少變動,不需要每次刷新都打 API),避免每次載入頁面都消耗 Admin API 配額。
3. `get_channels()` 回傳的 payload 加一個欄位 `attribution_model`(正規化後的簡短字串,如 `"data_driven"` / `"last_click"` / `"unknown"`),查詢失敗時容錯為 `"unknown"`,**不能因為這個查詢失敗就讓整個渠道對照卡片掛掉**(既有模組容錯慣例,比照到達頁分項統計失敗不影響主表格)。

**驗收標準**:對已知為 Data-driven 的 property 呼叫 `get_channels`,payload 含 `attribution_model: "data_driven"`;Admin API 呼叫失敗時(模擬例外)整個端點仍正常回傳其餘欄位,只是 `attribution_model` 為 `"unknown"`;24 小時內重複呼叫不重打 Admin API(驗證快取生效)。

**風險/回滾**:純新增唯讀查詢,不影響既有 conversions 抓取邏輯;單獨回滾只需移除新欄位,不影響既有渠道對照功能。

**驗收結果(2026-07-15 完成)**:
- `GA4Client` 新增 `get_attribution_settings()`;實際查證 GA4 Admin API v1alpha 的 enum 值為 `PAID_AND_ORGANIC_CHANNELS_DATA_DRIVEN` / `PAID_AND_ORGANIC_CHANNELS_LAST_CLICK` / `GOOGLE_PAID_CHANNELS_LAST_CLICK`(與原規劃預想的 `CROSS_CHANNEL_*` 命名不同,已在實作時對齊真實 SDK)。
- `insights_service.py` 新增 `_get_attribution_model()`:以 `kind="attribution_settings"`、固定 sentinel `date="static"` 存入既有 `ga4_insights_snapshots`(單例快取,不新增資料表),24 小時內命中快取不重打 Admin API;查詢失敗時優先回退「上一次快取到的值」而非直接判 unknown(設定極少變動,舊值比空值更有參考價值),完全沒有快取時才回退 `"unknown"`。
- `get_channels()` payload 新增 `attribution_model` 欄位。
- 新增 3 項單元測試(`test_ga4_channel_dimension.py`):無憑證時回退 unknown、正規化+快取生效(重複呼叫不重打 API)、快取過期但 API 失敗時回退舊值。
- 驗證:`python -m py_compile` 通過;GA4 相關 pytest 43 passed(含既有渠道對照/insights 測試,無回歸)。

**風險/回滾**:純文案調整,無邏輯風險;單檔回滾。

**驗收結果(2026-07-15 完成)**:

### 任務 2.2 — AI 白話解讀 prompt 用語同步調整

**變更檔案**:`backend/ai_service.py`(約 190-193、219 行的 `daily_channel` 相關 prompt 片段)

**實作步驟**:
1. 現有 prompt 把「收單」一律類比為「客人結帳前臨門一腳」(即 last-click 語意),與 Data-driven 屬性的實際意義不符。調整為中性描述:「收單數字代表這個管道在轉換路徑中拿到的功勞份量(依帳戶歸因設定而定),開發數字代表帶進第一批新客的量」,避免對 Data-driven 屬性做出「最後一步」的誤導性比喻。
2. 若 payload 已有 `attribution_model` 欄位(第一波產出),一併放進 AI prompt context,讓 AI 解讀時知道當下口徑,文案可依模式微調用字。

**驗收標準**:人工核對新 prompt 產出的白話解讀不再對 Data-driven 屬性使用「結帳前臨門一腳」這類 last-click 專屬比喻。

**風險/回滾**:純 prompt 文字調整,不影響既有 AI 解讀呼叫流程與其他 kind(intraday/landing_page/item)的 prompt;單獨回滾。

**驗收結果(2026-07-15 完成)**:
- `data.get("attribution_model")`(僅 `kind == "daily_channel"` 時讀取)對映出 `attribution_close_metaphor` 三種文字:`data_driven`→「主攻渠道是系統判斷在整趟購物旅程中貢獻比較大的（不是只看最後一次點擊）」、`last_click`→ 維持「主攻渠道是客人結帳前最後一次點擊進來的」、缺欄位/其他值→ 中性版本「系統判斷對這次購買貢獻比較大的（不一定是最後一次點擊）」。
- `kind_focus["daily_channel"]` 與術語直出範例(原「助攻/主攻」那條 bullet)都改用這個動態變數,不再寫死 last-click 比喻。
- 新增 2 項單元測試(`test_ga4_insights_wave2.py`):`data_driven` 情境斷言 prompt 不含「結帳前推最後一把/臨門一腳」字樣、`last_click` 情境斷言仍保留「最後一次點擊進來的」,避免未來改動又把兩種口徑寫混。
- 驗證:`python -m py_compile ai_service.py` 通過;GA4 相關 pytest 101 passed(含既有 `daily_channel` prompt 測試,無回歸)。

**風險/回滾**:行為變更(部分小樣本渠道的標籤從 assist/close/balanced 改為資料不足),但不影響其餘功能;`CHANNEL_MIN_SAMPLE` 可用 env 快速調整或設回 0(等同關閉此修正);單獨回滾。

## 五、明確不做

1. **完整使用者層級路徑重建與前期/中期/後期接觸點功勞分配**:需要 GA4 BigQuery Export,docs/21 已列為 Non-goals、另案評估,本計劃不做。
2. **改變 `CHANNEL_DIMENSION_MAP` 的資料來源或渠道對照的核心兩點式近似架構**:本計劃只調整門檻/文案/揭露,不重新設計比例公式本身——查證 Data-driven 屬性下,現有公式其實已經比預期更接近路徑歸因精神,不需要為了「更貼近路徑歸因」而大改公式。
3. **跨 property 批次盤點所有客戶的歸因模式**:第一波做的是「當前查詢的 property 即時查一次並快取」,不做背景批次掃描所有已連結 property 的歸因設定(用量低、非急迫需求,需要時可另案追加)。
