# 06_部署指南 (Deployment Guide)

## ☁️ 部署目標：Zeabur 雲端平台
**站略 (Site-tegy)** 針對 [Zeabur](https://zeabur.com/) 平台進行了優化，支援自動化的後端部署、前端靜態託管整合以及 PostgreSQL 資料庫連接。

---

## 🛠️ 前置準備
在開始部署前，請確保您已具備以下資源：
1. **GitHub 儲存庫**：將專案代碼推送到您的 GitHub。
2. **Google Cloud Console**：已建立 OAuth 2.0 Client ID，並將 Zeabur 的網址加入「授權的重新導向 URI」。
3. **Facebook Developers**：已建立 App，並取得 App ID 與 App Secret。

---

## 🚀 部署步驟

### 1. 資料庫配置 (PostgreSQL)
1. 在 Zeabur 建立一個 **Postgres** 服務。
2. 複製 **Private Connectivity** (內網連線) 字串（例如 `postgresql://root:password@postgresql.zeabur.internal:5432/postgres`）。

### 2. 後端部署 (Backend)
1. 在 Zeabur 建立服務，選擇您的 Git 儲存庫。
2. **Root Directory**: 設定為 `backend`。
3. **Networking**: 確認 Port 為 `8000`。
4. **環境變數 (Variables)**：
   - `DATABASE_URL`: 填入步驟 1 的 PostgreSQL 內網字串。
   - `GOOGLE_CLIENT_ID`: 您的 Google Client ID。
   - `GOOGLE_CLIENT_SECRET`: 您的 Google Client Secret。
   - `ENCRYPTION_KEY`: 您的 Fernet 加密金鑰（可透過 `cryptography.fernet.Fernet.generate_key()` 產生）。
   - `SUPER_ADMIN_EMAIL`: 您的管理員 Email (支援多個，以逗號分隔)。
   - `ENABLE_REPORT_SCHEDULER`: 若後端同時負責排程，設為 `true`；若另建專用 scheduler worker，web service 請設為 `false`。

### 2.1 週報排程部署建議
目前週報自動排程使用 APScheduler。若直接掛在 Web Service 內，當雲端平台讓 instance 休眠、重啟或縮容時，排程時間點可能完全錯過，因此會出現：

- 沒有任何「觸發」log
- `next_run` 已過期但沒有產出報表
- 服務恢復後也沒有自動補跑

目前程式已補上兩個保護機制：

- 啟動時會記錄 scheduler 狀態與載入的 jobs
- 啟動後會對 `next_run` 已過期的排程做一次 catch-up 補跑

若要在 Zeabur 上穩定執行，建議再建立一個 **專用 scheduler worker**：

1. 新增一個服務，**Root Directory** 同樣設為 `backend`。
2. **Start Command** 設為 `python scheduler_worker.py`。
3. 將此 worker 的 `ENABLE_REPORT_SCHEDULER` 設為 `true`。
4. 原本 Web Service 的 `ENABLE_REPORT_SCHEDULER` 設為 `false`，避免同一份排程被多個程序重複執行。

若您暫時不拆 worker，也至少要確保 backend instance 不會進入休眠；否則精準到時觸發這件事，單靠 web process 仍然不可靠。

### 3. 前端部署 (Frontend)
1. 建立服務，選擇相同的 Git 儲存庫。
2. **Root Directory**: 設定為 `frontend`。
3. **Build Command**: `npm install && npm run build`。
4. **Output Directory**: `dist`。
5. **環境變數 (Variables)**：
   - `VITE_API_URL`: **後端服務的公網網址** (不含末尾的 `/api`)。
   - `VITE_GOOGLE_CLIENT_ID`: 與後端一致。

## 🔄 資料庫遷移 (Migrations)
站略使用 Alembic 進行資料庫管理。在 Zeabur 上，您可以透過服務的 **Console (控制台)** 手動執行：

```bash
cd backend
alembic upgrade head
```
*註：系統在啟動時會嘗試自動同步 `super_admin_email` 權限。*

---

## 🛡️ 維護與除錯
- **日誌檢查**：若 API 無法連線，請優先檢查 Zeabur 的 Runtime Logs。
- **排程檢查**：可打 `/health`，查看 `checks.scheduler` 是否顯示 `enabled=true`、`running=true` 與目前 `job_count`。
- **CORS 議題**：若前端發生跨網域錯誤（Access-Control-Allow-Origin），請：
  1. **檢查後端環境變數**：確保後端的 `ALLOWED_ORIGINS` 包含前端的完整網址（例如 `https://datavue-frontend.tabisme.com`）。
  2. **正則匹配 (Regex)**：目前代碼已支援自動匹配 `*.tabisme.com` 與 `*.zeabur.app`。若更換全新域名，請修改 `main.py` 中的 `allow_origin_regex`。
  3. **設定位置**：此設定必須在 **Zeabur 後端的 Variables** 中修改，而非前端。
- **Token 過期**：確保 `ENCRYPTION_KEY` 保持不變，否則資料庫中的加密 Token 將無法解密。
