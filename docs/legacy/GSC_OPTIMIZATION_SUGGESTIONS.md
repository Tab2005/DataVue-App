# GSC 模組資料抓取優化建議

本文檔旨在針對 `backend/gsc_service.py` 中的 `get_analytics` 函式提出優化建議，以解決目前存在的 25,000 筆資料抓取上限問題，並提升整體效能。

## 當前問題分析

目前的 `get_analytics` 函式雖然實作了基本的分頁邏輯，但存在一個關鍵限制：程式碼中硬性設定了 `max_rows = 25000` 的上限。

這導致即使 Google Search Console (GSC) API 透過分頁可以提供超過 25,000 筆的總資料量，程式也會在達到此上限時提前終止抓取。這是對 GSC API 限制的誤解，API 的限制是**單次請求**最多 25,000 筆，而非總資料量。

## 優化前後行為比較

### 優化前（目前行為）

目前的程式會一直抓取資料，直到發生以下三種情況之一才停止：

1.  **資料已全部抓完：** 如果請求的總資料量小於 25,000 筆（例如 5,000 筆），程式會抓完這 5,000 筆後正常停止，並回傳**完整**資料。
2.  **達到 25,000 筆上限：** 如果總資料量超過 25,000 筆（例如 40,000 筆），程式會在抓到第 25,000 筆時**強制停止**。此時回傳的資料是**不完整**的。
3.  **API 回傳空資料：** 在抓取過程中，如果下一次請求沒有回傳任何資料，也會停止。

**結論：** 目前最大的問題是，當資料總量超過 25,000 筆時，系統回傳的數據會被截斷，導致資料不完整。

### 優化後（建議行為）

採用優化建議（移除 `max_rows` 限制並提高 `batch_size`）後，系統將能保證抓取到完整的數據。

1.  **持續抓取：** 程式會以 25,000 筆為單位持續發送請求。
2.  **智慧判斷終點：** 當某一次 API 回傳的資料筆數**小於** 25,000 筆時，程式會將其視為最後一頁，然後停止抓取。
3.  **保證資料完整性：** 無論總資料量是 5,000 筆、50,000 筆還是 500,000 筆，系統都能確保抓取到每一筆資料，徹底解決數據被截斷的問題。

**結論：** 優化後，系統將能正確處理 GSC API 的分頁邏輯，確保任何情況下都能獲得**完整**的數據集。

## 優化建議

以下是具體的優化建議，旨在解決上述問題並提升效能。

### 1. 修正分頁邏輯並移除硬性上限

這是解決問題的核心步驟。必須移除 `max_rows` 的硬性限制，並讓迴圈能夠持續抓取，直到所有資料頁都已讀取完畢。

**建議作法：**

修改 `while` 迴圈的判斷條件，讓它只在 API 回傳資料為空，或回傳筆數小於請求筆數時才終止。

```python
# 建議的迴圈邏輯
while True:
    request = {
        'startDate': start_date,
        'endDate': end_date,
        'dimensions': dimensions,
        'rowLimit': batch_size,
        'startRow': start_row
    }
    
    response = service.searchanalytics().query(siteUrl=site_url, body=request).execute()
    rows = response.get('rows', [])
    
    # 如果 API 沒有回傳任何資料，代表已經沒有更多內容
    if not rows:
        break
        
    all_rows.extend(rows)
    
    # 如果回傳的筆數小於請求的筆數，代表這是最後一頁
    if len(rows) < batch_size:
        break
        
    start_row += batch_size
```

### 2. 提高單次請求的資料筆數 (Batch Size)

目前的程式碼以 `1000` 筆為一個批次進行抓取。為了最大化效率、減少 API 請求次數，應將此數值提高至 GSC API 允許的上限。

**建議作法：**

將 `batch_size` 變數的值從 `1000` 提高到 `25000`。

```python
# 建議修改
batch_size = 25000  # 原本的值是 1000
```

這項修改將大幅減少 API 呼叫次數。例如，抓取 100,000 筆資料，原本需要 100 次 API 請求，修改後僅需 4 次。

### 3. 整合快取機制 (Caching)

對於參數完全相同的查詢（例如，同一個網站、相同的日期範圍與維度），在短時間內可能會被使用者重複請求。實作快取可以將首次查詢的結果暫存起來，後續的相同請求可以直接從快取中快速回傳，避免不必要的 API 呼叫，從而節省配額並提升使用者體驗。

**建議作法：**

利用專案中已有的 `backend/cache.py` 模組。可以設計一個以查詢參數（`site_url`, `start_date`, `end_date`, `dimensions`）組合而成的快取鍵（Cache Key），並將查詢結果作為值（Value）存入，同時設定一個合理的過期時間（例如 1 小時或 1 天）。

## 總結

綜合以上建議，立即可以執行的關鍵步驟為：

1.  **修正分頁邏輯**：移除 `max_rows` 限制，確保能抓取所有資料。
2.  **提高批次大小**：將 `batch_size` 調整為 `25000`，減少 API 請求次數。

長期來看，再結合**快取機制**，可以建構一個更高效、更具擴展性的 GSC 資料服務。

## Redis 快取方案（本地開發 vs 佈署）

### 目前快取機制的限制

現有的快取為記憶體內的 `TTLCache`，只存在單一後端進程：

- **不共享**：多實例部署時，快取無法跨容器共用。
- **不持久**：服務重啟即清空。
- **時間敏感**：TTL 到期就會重新打 GSC API。

這會導致佈署到網路端時，即使資料相同，仍可能頻繁 cache miss，前端卡在「正在從 Search Console 抓取數據」。

### Redis 對本地與網路的影響

| 環境 | 影響 | 結論 |
|---|---|---|
| 本地端 | 可選擇不啟用 Redis，使用原本記憶體快取 | 不影響開發流程 |
| 佈署端 | Redis 讓快取可跨實例共享並持久 | 顯著降低 API 延遲 |

**重點**：Redis 主要解決佈署端「多實例 / 重啟 / cache miss」問題，對本地端影響小，可用環境變數切換。

### 建議做法（不改動前端）

1. **後端快取抽象化**：
    - 若 `REDIS_URL` 存在 → 使用 Redis
    - 若不存在 → 回退至原本的 `TTLCache`

2. **設定 REDIS_URL（佈署端）**：
    - 在 Zeabur 建立 Redis 服務後，取得連線字串
    - 將連線字串設定到後端服務的環境變數 `REDIS_URL`
    - 常見格式：
      - `redis://:<password>@<host>:<port>/0`
      - 若有內網連線，優先使用內網位址

3. **設定 REDIS_URL（本地端）**：
    - 本地端可不設定 `REDIS_URL`（回退 TTLCache）
    - 若需測試 Redis：在 `backend/.env` 加入 `REDIS_URL`

4. **快取鍵一致化**：
    - `gsc_analytics:{user_id}:{site_url}:{start_date}:{end_date}:{dimensions}`

5. **TTL 策略**：
    - 網路端建議 10–30 分鐘（視資料更新需求）
    - 本地端可維持 2 分鐘

### Redis vs DB 快取取捨

- **Redis（推薦）**：共享、快、支援 TTL、對佈署端效果最顯著。
- **DB 快取（備案）**：可持久，但效能較慢，DB 壓力高，TTL 管理麻煩。

### 何時採用 Redis

- 佈署端為多實例（或可能 auto-scaling）
- 目前前端卡在載入動畫，且後端每次都打 GSC API
- 希望快取能跨重啟持久存在

若你確認採用 Redis，可先只針對 GSC 快取替換，風險最小、效果最大。

## 只替換 GSC 快取到 Redis：實作計劃（待確認）

> 目標：只將 GSC `get_analytics` 的快取從記憶體 TTLCache 改為 Redis，其他模組維持不變。

### A. 需求確認

- Zeabur 已建立 Redis 服務
- 後端可取得 `REDIS_URL`
- GSC 快取需可共享（跨實例 / 重啟仍可命中）

### B. 實作步驟

1. **新增 Redis 依賴**
    - 後端新增 `redis` 或 `redis[async]` 套件

2. **建立 Redis 快取介面（僅 GSC 使用）**
    - 新增 Redis cache helper（例如 `backend/redis_cache.py`）
    - 功能：`get_cached` / `set_cached` / `invalidate`（只針對 GSC）

3. **GSC 快取接入點**
    - 修改 `backend/gsc_service.py` 的 `get_analytics`
    - 讀取順序：Redis → 若無，使用 GSC API → 回寫 Redis

4. **環境變數切換策略**
    - 若 `REDIS_URL` 不存在 → 回退到 TTLCache
    - 本地端可不啟用 Redis

5. **TTL 策略**
    - 佈署端：10–30 分鐘
    - 本地端：維持 120 秒

6. **記錄與監控（可選）**
    - 增加 log：`[GSC REDIS HIT]` / `[GSC REDIS MISS]`
    - 方便比對佈署前後命中率

### C. 驗證步驟

1. 佈署後第一次請求（MISS）
2. 相同參數第二次請求（HIT）
3. 切換分頁後再次返回（應為 HIT）

### D. 回復方案

- 移除 `REDIS_URL` 即回退到 TTLCache
- 或直接切回舊版 `gsc_service.py`

### E. 影響範圍

- 僅 GSC 模組快取行為改變
- 其他模組（FB/GA4）不受影響

## 近期佈署效能結果（已實作）

以下為近期在佈署端實作並觀察到的效能改善，與「關鍵字分析載入卡住」問題直接相關：

1. **關鍵字分析改為後端分頁載入**
    - 針對 `dimensions=query` 使用 `limit` + `offset` 逐頁讀取
    - 前端提供「載入更多資料」，避免一次拉取 18k+ 大量資料
    - 減少 JSON 序列化與前端渲染阻塞

2. **啟用 GZip 壓縮**
    - 後端加入 `GZipMiddleware`
    - 大型 JSON 回應傳輸時間顯著縮短

**觀察結果：**
- 佈署端頁面載入速度明顯加快
- 前端「正在從 Search Console 抓取數據」卡住的情況明顯降低
