## 零、評估方法與結論總覽

拆分優先級不是只看行數，而是「**行數 × 變更熱度 × 內部結構證據**」三者綜合：

- **行數**：docs/31 的統計已過時（例如 `repository.py` 文件記載 2880 行，現況已 3205 行），本文件全部採用 2026-07-14 重新量測的現況數字。
- **變更熱度**：以 `git log --since=2026-05-01` 統計近兩個半月每檔的 commit 次數。大而冷的檔案（如 `GSCStats.jsx` 3717 行但只改過 2 次）拆分收益低、風險高；大而熱的檔案（如 `repository.py` 58 次）每次改動都要在三千行裡找位置、AI 輔助開發時每次都要吃掉大量 context，拆分收益最高。
- **結構證據**：實際檢視每個候選檔案的 class/function 分佈，確認存在自然的拆分邊界（方法群聚），而不是硬拆。

## 二、共同拆分原則（每一波都適用）

1. **行為保持（behavior-preserving）**：只做「搬移 + re-export」，不改任何函式簽名、不改邏輯、不改例外類型。每一波拆完，既有測試必須以「修改前後失敗集合完全一致」為基準（目前 `test_meta_andromeda_module.py` 有 29 個已知環境性失敗，拆分不得增減）。
2. **Facade 模式保住既有 import 路徑**：後端拆分一律採「單檔 → 同名 package」手法：`repository.py` → `repository/` 目錄 + `__init__.py` re-export 原有公開名稱。所有 `from .repository import MetaAndromedaRepository` 的既有呼叫端**零修改**。
3. **一波一 commit，拆分 commit 不夾帶任何功能修改**：讓 review 時 `git diff --stat` 只看到搬移，方便用 `git log --follow` 追歷史。
4. **先拆熱的，冷的緩拆**：P0 檔案幾乎每個功能案都會碰，先拆立即降低後續所有開發的成本；P3 的 GSCStats/GA4Stats 雖然最大，但近期沒人動它，收益排最後。
5. **前端拆分以「已存在的目錄慣例」為家**：`components/Analytics/`、`components/Reports/`、`components/Landing/` 已是既有慣例，新拆出的元件比照建立 `components/MetaAndromeda/`、`components/GA4Insights/`、`components/Contribution/` 等子目錄。
6. **每波拆完跑對應驗證**：後端跑 `pytest` 對應測試檔 + `python -c "import ..."` 冒煙；前端跑 `npm run build` + 既有 vitest 測試（`ContributionAnalysis.test.jsx` 等 5 個頁面測試是現成的回歸保護網）。

## 十二、風險與緩解

| 風險 | 緩解 |
|---|---|
| repository/service 拆分期間有功能案並行，產生大範圍衝突 | 每波控制在 1-2 天內完成合併；波次啟動前確認無進行中的 meta_andromeda 功能分支 |
| runtime.py 被 worker 共用，拆錯 import 造成 worker 靜默降級到 heuristic | 拆後必跑 `meta_andromeda_shared_runtime_smoke.py`；部署後檢查 worker 日誌無 import error，並抽查一筆評分確認 `scoring_mode=ai` |
| scheduler 拆分破壞 job 註冊順序或 SERVICE_ROLE 分工 | engine 單例唯一化；對照 docs/24/25 的 worker 架構筆記逐 job 核對 |
| 前端拆分遺失 usePermission 守門 | 每個拆出的面板保留原有權限判斷；對照 memory 中 require_permission 的複合守門原則 |
| mixin 組裝讓方法定義位置變難找 | `__init__.py` 中為每個 mixin 加一行註解標明領域；IDE 的 go-to-definition 不受影響 |
