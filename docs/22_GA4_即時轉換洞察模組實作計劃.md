## 零、背景與功能對照

### 0.1 目標

把系統的 GA4 能力從「隔日報表查詢」升級為「當日洞察 + 主動示警」，對標市售服務的核心賣點：解決 GA4 介面延遲、異常主動通知、渠道/到達頁/商品的轉換視角分析。**排除項**：同業流量來源分佈（需跨客戶專有資料池，非技術問題，不做）。

### 0.2 功能對照與可行性（2026-07-06 評估）

| 服務賣點 | 技術本質 | 系統現況 | 本計劃對應 |
|---|---|---|---|
| 解決 GA4 延遲、即時掌握 | GA4 Data API `runRealtimeReport` + 當日 intraday 查詢 | 只用標準 RunReport（隔日） | 第 2 波：當日儀表板 |
| 數據異常立即通知 | 統計基線 + 排程偵測 + 推播 | 零件全有：APScheduler、LINE push（`send_line_push_message`）、Email | 第 1 波：異常偵測（**優先，CP 值最高**） |
| 儀表板每週自動更新 | 排程報表 | GA4 週報已存在 | 第 3 波：摘要整合 |
| 渠道轉換、主攻/助攻趨勢 | 首次觸點 vs 最後觸點對照 | 有渠道維度查詢 | 第 2 波：渠道對照分析 |
| 管道貢獻度與預算建議 | 貢獻模型 | **docs/21 contribution 模組（更深）** | 不重做，頁面互連 |
| 商品潛力識別 | GA4 電商 item 報表 | 未做 | 第 2 波 |
| 到達頁轉換/跳出率 | landingPage 維度 | 未做 | 第 2 波 |
| KPI 制定與追蹤 | 目標設定 + 進度 | 未做 | 第 3 波 |

### 0.3 誠實的技術邊界（先講清楚，避免過度承諾「即時」）

- **Realtime API**：只有近 30 分鐘窗口、維度/指標集有限（無電商 item 細節、無 landingPage），適合做「現在有多少活躍使用者/事件」的心跳窗格，不能撐起完整儀表板。
- **Intraday（當日）資料**：標準 GA4 property 的當日資料可查但有處理延遲（數分鐘到數小時不等，Google 不保證 SLA）。本計劃的「即時」定義為：**當日資料每小時自動刷新 + 近 30 分鐘 Realtime 心跳**，比 GA4 介面慣性的隔日解讀快 24-48 小時，但不是秒級即時。
- **API 配額**：GA4 Data API 有 property 級配額（token 制）。每小時刷新 × 多帳戶會消耗配額，需以 DB 快取 + 只刷新「有開啟洞察頁或有異常規則」的 property 控制用量。

### 3.1 anomaly.py — 異常偵測規格（第 1 波核心，純函數）

**基線模型**（對抗星期效應與緩慢趨勢，不引入重依賴）：
- 取過去 8 週「同星期」的該指標值（daily）或「同星期同累計時段」值（hourly，當日 00:00 至當前小時的累計），至少 4 個樣本才啟用。
- 基線中心 = 中位數；離散度 = MAD（中位數絕對偏差）。
- 告警條件：`observed < median - k*MAD`（drop）或 `> median + k*MAD`（spike），k 由 sensitivity 決定（high=2 / medium=3 / low=4）；超出 2k 為 `critical`。
- MAD = 0 的退化情況（樣本全同值）以 `max(MAD, 0.05 * median)` 下限保護。
- 冷卻：同 rule 在 `cooldown_hours` 內不重複告警；恢復正常時發一次「已回穩」通知（critical 事件限定）。

**通知文案**（LINE 純文字，走既有 `send_line_push_message`）：
```
⚠️ GA4 異常（{property 名}）
{指標中文名} 今日累計 {observed}，低於預期區間 {low}~{high}（過去 8 週同時段基線）
可能方向：檢查廣告投放是否中斷 / 網站追蹤碼 / 檔期效應
{系統網址}/ga4-insights
```

**排程**（`core/scheduler.py` 比照既有 job 模式新增）：
- `ga4_insights_intraday_job`：每小時整點 +10 分（避開 GA4 處理延遲尖峰），對「有啟用 hourly 規則或近 7 天有人開過洞察頁」的 property 刷新 intraday snapshot → 跑 hourly 規則評估。
- `ga4_insights_daily_job`：每日 09:00（`SCHEDULER_TIMEZONE`），刷新昨日 daily snapshot → 跑 daily 規則評估。

### 3.2 當日儀表板資料（第 2 波）

- **Intraday 卡**：`RunReport(date=today)`，指標 sessions / conversions / purchaseRevenue / 主要事件數，維度 hour + sessionDefaultChannelGroup；與「過去 8 週同星期同時段累計基線」並列畫走勢（超出區間即異常視覺化，與告警同一套函數）。
- **Realtime 心跳卡**：`runRealtimeReport`（activeUsers、近 30 分鐘事件數 by unifiedScreenName/country），標示「近 30 分鐘」窗口，不與 intraday 混排。
- 前端一律讀 `ga4_insights_snapshots` 快取（intraday 每小時由 job 更新；頁面提供「立即刷新」按鈕，rate-limit 每 property 10 分鐘一次）。Realtime 心跳例外：即打即查（輕量、單獨配額池）。

### 3.3 渠道主攻/助攻對照（第 2 波）

同一期間跑兩份口徑並排：
- **收單視角**：`sessionDefaultChannelGroup` × conversions（最後接觸歸因近似）
- **開發視角**：`firstUserDefaultChannelGroup` × conversions（首次接觸歸因）
- 逐渠道計算「開發/收單比」：>1.3 標「助攻型（上游開發）」、<0.7 標「主攻型（下游收單）」，其餘「均衡」。
- 頁面註明這是 GA4 兩種歸因口徑的對照近似，並放「深度增量分析 → 貢獻分析頁」連結（docs/21 模組）；Meta 渠道的收割判定以貢獻分析的歸因通膨指數為準。

### 3.4 到達頁與商品分析（第 2 波）

- **到達頁**：`landingPage` × sessions / engagementRate / conversions / bounceRate，附「高流量低轉換」自動標記（sessions 前 25% 且轉換率後 25%）。
- **商品**：`itemName` × itemsViewed / itemsAddedToCart / itemsPurchased / itemRevenue，計算瀏覽→加購、加購→購買轉換率；「潛力商品」= 瀏覽成長快（近 7 天 vs 前 7 天）且加購率高於中位數但曝光（views）仍低於中位數。
- 皆為 daily snapshot（每日 job 一併抓），非 intraday。

### 3.5 API 端點（前綴 `/api/ga4/insights`，掛 `require_module("ga4")`，**不動既有 `/api/ga4/*`**）

| 方法 | 路徑 | 說明 |
|---|---|---|
| GET | `/dashboard?property_id=` | 當日儀表板（intraday 快取 + 基線區間） |
| POST | `/dashboard/refresh` | 手動刷新 intraday（rate-limited） |
| GET | `/realtime?property_id=` | Realtime 心跳（直查，輕量） |
| GET | `/channels?property_id=&days=` | 渠道主攻/助攻對照 |
| GET | `/landing-pages?property_id=&days=` | 到達頁分析 |
| GET | `/items?property_id=&days=` | 商品分析 |
| GET/POST/PUT/DELETE | `/anomaly-rules` | 異常規則 CRUD |
| GET | `/anomaly-events?property_id=` | 告警歷史（分頁）；PATCH `/anomaly-events/{id}/ack` 標記已讀 |

1. **資料流（同週報模式）**：前端組裝該頁籤 payload（含基線區間、主攻/助攻或高流量低轉換等標記結果、未 ack 告警摘要）→ `aiService.analyzeDataStream(data, context, 'ga4_insights', ...)` 串流顯示 → 完成後 `PUT /api/ga4/insights/snapshots/{id}/ai-summary` 持久化；重開頁面直接讀快取內的 `ai_summary`，不重打 AI。**intraday snapshot 被每小時 job 覆寫時 `ai_summary` 一併清空**（資料已變，舊解讀失效，按鈕回到「開始 AI 解讀」）。
