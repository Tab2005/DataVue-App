# GA4 網站週報功能實作計畫 (GA4 Weekly Report Implementation Plan)

## Objective
在不大幅改動現有 Facebook Ads 週報架構的前提下，擴充系統以支援 GA4 的網站成效週報。使用者可以透過現有的週報介面，建立、檢視、與排程自動產生 GA4 網站的成效摘要與趨勢。

## Tech Stack
- Backend: FastAPI, SQLAlchemy, Google Analytics Data API, apscheduler
- Frontend: React 19, Vite, React Router, React Query
- Database: PostgreSQL (via Alembic)

## Architecture Decisions
1. **複用現有模型**: `WeeklyReport` 與 `ReportSchedule` 模型結構已夠通用，將新增 `module_type` 欄位（預設 `fb_ads`，可為 `ga4`）以區分不同來源的週報。
2. **條件選擇器抽換**: 根據 `module_type`，前端精靈第一步動態切換：選擇廣告帳號 (FB) 或選擇 GA4 資源 (Properties)。
3. **數據獲取抽象化**: 在 `report_service.py` 內建立分流邏輯，根據 `module_type` 決定呼叫 `FB API` 或是 `GA4 API`，並統一回傳符合報表模板的 JSON 結構 (KPIs, trends, table_data)。
4. **離線自動化獲取**: 透過資料庫存儲的 `ga4_refresh_token`，讓背景排程能自動換取 `access_token`，不需使用者在線介入即可完成自動化週報。

  - Files: `backend/database/models/report.py`
- [ ] Task 2: 產生 Alembic 遷移檔
  - Description: 執行 alembic 指令產生並套用 migration。
  - Verify: 檢查資料庫表中是否出現 `module_type` 欄位且不影響舊資料。
- [ ] Task 3: 擴充 GA4 Service
  - Description: 在 `ga4_service.py` 實作取得 GA4 彙總 (summary) 和趨勢 (trends) 數據的函式，以相容週報需要的格式。指標將包含 `activeUsers`, `sessions`, `screenPageViews`, `engagementRate` 等。

### Checkpoint: Backend Data Ready
- [ ] `module_type` 欄位更新成功。
- [ ] 能透過 GA4 Service 取得週報所需格式的數據。

- [ ] Task 5: 更新 Router API
  - Description: 在 `routers/reports.py` 調整 CRUD API 的 Pydantic Schema (`ReportCreate` / `ScheduleCreate`)，使其支援 `module_type`。
