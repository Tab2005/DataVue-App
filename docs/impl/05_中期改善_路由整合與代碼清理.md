## M-1：routers/auth.py 整合 integration_service

### 問題

`routers/auth.py` 第 68–85 行的 `get_token_status` 端點仍直接讀取 `User` 表中即將廢棄的舊欄位（`fb_access_token`、`token_expires_at`），而非使用已建好的 `integration_service`。

```python
# 現有問題代碼（auth.py）
target = db.query(User).filter(User.google_id == user_id).first()
if target and hasattr(target, 'fb_access_token') and target.fb_access_token:
    token_exists = len(str(target.fb_access_token)) > 10

if not target or not target.token_expires_at:
    ...
```

**後果**：`UserIntegration` 遷移完成後，此端點的 `token_exists` 永遠為 `False`，導致前端誤判為未連線狀態。

---

### 修復方案

**檔案**：`backend/routers/auth.py`

#### 修改步驟

**1. 更新 import 區塊**

```python
# 在 auth.py 頂部 import 區塊新增：
from services.integration_service import (
    get_user_integration,
    get_decrypted_access_token,
)
```

**2. 修改 get_token_status 端點邏輯**

```python
@router.get("/token-status")
@limiter.limit("30/minute")
def get_token_status(
    request: Request,
    db: Session = Depends(get_db),
    current_user = Depends(get_current_user),
):
    """
    查詢目前使用者的 Facebook Token 狀態。
    
    改用 integration_service 查詢 user_integrations 表，
    不再直接讀取 User 表的舊欄位。
    """
    user_id = current_user.id

    # ✅ 新版：查詢 user_integrations 表
    integration = get_user_integration(db, user_id, "facebook")

    token_exists = False
    expires_at = None
    is_expired = True

    if integration and integration.access_token:
        token_exists = True
        expires_at = integration.token_expiry
        if expires_at:
            from datetime import datetime, timezone
            is_expired = datetime.now(timezone.utc) > expires_at
        else:
            is_expired = False  # 無過期時間視為有效

    return {
        "token_exists": token_exists,
        "expires_at": expires_at.isoformat() if expires_at else None,
        "is_expired": is_expired,
        "provider": "facebook",
    }
```

---

### 驗收標準

- [ ] `routers/auth.py` 不再直接讀取 `User.fb_access_token`
- [ ] `routers/auth.py` 不再直接讀取 `User.token_expires_at`  
- [ ] `/api/auth/token-status` 端點在沒有 `user_integrations` 記錄時正確回傳 `token_exists: false`
- [ ] `/api/auth/token-status` 端點在有 `user_integrations` 記錄時正確回傳 `token_exists: true`
- [ ] 對應測試已更新（`tests/test_auth.py`）

## M-4：清理 dependencies.py 的 logging.basicConfig

### 問題

`backend/dependencies.py` 第 7 行呼叫 `logging.basicConfig(level=logging.INFO)`，可能在 `core/logging.py` 的 `setup_logging()` 之前執行，導致根 Logger 上存在未格式化的 StreamHandler，與統一日誌設定衝突。

```python
# dependencies.py 第 7 行（有問題）
import logging
logging.basicConfig(level=logging.INFO)  # ← 應移除
```

---

### 修復方案

**檔案**：`backend/dependencies.py`

#### 修改步驟

**1. 移除 basicConfig 呼叫**

```python
# 舊版（移除前）：
import logging
logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

# 新版（移除後）：
import logging
logger = logging.getLogger(__name__)
# basicConfig 由 core/logging.py 的 setup_logging() 統一設定
```

**2. 確認 setup_logging() 在 main.py 中先於路由 import 執行**

```python
# main.py 應確保以下順序：
from core.logging import setup_logging
setup_logging()  # 第一步，設定 logger

# 然後才 import 路由（間接觸發 dependencies.py 的 import）
from routers import auth, users, ...
```

---

### 驗收標準

- [ ] `dependencies.py` 中不含 `logging.basicConfig()`
- [ ] 應用程式啟動後日誌格式一致（有時間戳、模組名稱）
- [ ] `logger = logging.getLogger(__name__)` 保留（dependencies.py 仍可記錄日誌）

## L-1：統一 get_db() 定義

### 問題

`get_db()` 在兩處各自獨立定義，功能完全相同：

| 位置 | 用途 |
|------|------|
| `backend/database/engine.py` 第 81 行 | 正式定義，由 `database/__init__.py` 匯出 |
| `backend/dependencies.py` 第 16 行 | 舊版遺留，功能重複 |

若兩地的 `SessionLocal` 設定不同步（例如未來修改連線池），可能產生不一致的行為。

---

### 修復方案

**目標**：所有路由統一使用 `from database import get_db`，廢棄 `dependencies.get_db`。

#### Step 1：確認受影響的 import

```powershell
# 找出所有從 dependencies 匯入 get_db 的地方
Select-String -Path "backend/**/*.py" -Pattern "from dependencies import.*get_db|dependencies\.get_db" -Recurse
```

#### Step 2：批量替換 import

```python
# 將所有這種模式：
from dependencies import get_db

# 替換為：
from database import get_db
```

#### Step 3：從 dependencies.py 移除重複定義

```python
# 在 dependencies.py 中，將：
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

# 替換為重新匯出（維持向後相容，但避免重複定義）：
from database import get_db  # 將此加入 dependencies.py 的 import 區塊

# 並加上廢棄警告：
import warnings
def get_db_deprecated():
    """
    已廢棄：請改用 from database import get_db
    此函式保留僅為向後相容，將於下一版本移除。
    """
    warnings.warn(
        "dependencies.get_db is deprecated, use 'from database import get_db' instead",
        DeprecationWarning,
        stacklevel=2,
    )
    from database import get_db as _get_db
    yield from _get_db()
```

---

### 驗收標準

- [ ] 所有路由中 `Depends(get_db)` 的來源為 `database` 套件
- [ ] `dependencies.py` 不再有獨立的 `get_db()` 函式定義
- [ ] `from database import get_db` 正確匯出（無 ImportError）

## L-3：ga4_service.py 更新 cache API

### 問題

`backend/ga4_service.py` 第 28–29 行仍使用舊版快取函式名稱：

```python
from cache import generate_cache_key, get_cached, set_cached, analytics_cache
from redis_cache import get_cached_redis, set_cached_redis
```

雖然 `cache.py` 提供向後相容包裝器（函式仍可正常呼叫），但語意已改變，可能帶來困惑。

---

### 修復方案（下次重構 ga4_service.py 時一併處理）

```python
# 舊版（向後相容包裝器）
from cache import generate_cache_key, get_cached, set_cached, analytics_cache
from redis_cache import get_cached_redis, set_cached_redis

# 新版（直接使用統一 cache API）
from cache import (
    generate_cache_key,
    cache_get,           # 替換 get_cached / get_cached_redis
    cache_set,           # 替換 set_cached / set_cached_redis
    analytics_cache,     # 可繼續使用（L1 快取實例）
)
```

**注意**：此修改需同時更新 `ga4_service.py` 中所有 `get_cached()`、`set_cached()`、`get_cached_redis()`、`set_cached_redis()` 的呼叫，確認語意等同後才能替換。
